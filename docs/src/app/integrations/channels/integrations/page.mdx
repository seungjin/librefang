# Integrations & Protocol Channels

Messaging platform integrations, protocol adapters, and webhook-based channels.

---

## LINE

LINE now ships as the `librefang.sidecar.adapters.line` Python sidecar (the in-process Rust adapter was removed in the sidecar migration). The sidecar runs **its own HTTP webhook server**, so the URL you register at the LINE Developers Console must point at the sidecar host/port, not the LibreFang API port.

### Prerequisites

- A LINE Messaging API channel (from the [LINE Developers Console](https://developers.line.biz/en/docs/messaging-api/))
- The `librefang` Python SDK installed where the sidecar will run
- A publicly accessible webhook URL pointing at the sidecar (typical pattern: an HTTPS reverse proxy in front of the sidecar's `LINE_WEBHOOK_PORT`)

### Setup

1. Go to the [LINE Developers Console](https://developers.line.biz/console/) and create a new Messaging API channel.
2. Note the **Channel Secret** and issue a **Channel Access Token**.
3. Set both secrets in `~/.librefang/secrets.env` (or via the dashboard's "Set API Key" button, which writes the same file):

```bash
LINE_CHANNEL_SECRET=your-secret
LINE_CHANNEL_ACCESS_TOKEN=your-access-token
```

4. Declare the sidecar in `config.toml`:

```toml
[[sidecar_channels]]
name = "line"
command = "python3"
args = ["-m", "librefang.sidecar.adapters.line"]
channel_type = "line"
default_agent = "assistant"
[sidecar_channels.env]
LINE_WEBHOOK_PORT = "9090"
# LINE_WEBHOOK_PATH = "/webhook"
# LINE_ACCOUNT_ID = "production"   # optional, multi-bot routing key
```

5. In the LINE Developers Console, set the webhook URL to your sidecar host (e.g., `https://your-sidecar-host:9090/webhook`).
6. Restart the daemon — the supervisor launches the sidecar, which validates the access token via `GET /v2/bot/info` at startup and then accepts inbound webhook POSTs.

### How It Works

The sidecar runs as a supervised subprocess of LibreFang. It serves a standalone HTTP webhook endpoint on `LINE_WEBHOOK_PORT` (default `9090`) and `LINE_WEBHOOK_PATH` (default `/webhook`), verifies every inbound POST's `X-Line-Signature` HMAC-SHA256 over the raw request body (missing → 400, mismatched → 401), and forwards parsed `message` events to the kernel via newline-delimited JSON-RPC over stdio. Outbound responses go through `POST /v2/bot/message/push` with the Bearer access token; 429 responses honour `Retry-After` and chunk-level fail-open keeps a single throttled chunk from dropping the rest of a multi-part reply.

## DingTalk

Supports two modes:
- **Stream mode** (default, recommended): WebSocket long-connection via the DingTalk Stream protocol. No public IP or port required.
- **Webhook mode** (legacy): HTTP callback server. Requires a public-facing URL.

### Prerequisites

- A DingTalk application from the [DingTalk Open Platform](https://open.dingtalk.com/document/)
- For **stream mode**: App Key and App Secret from your DingTalk application
- For **webhook mode**: A custom robot with webhook access token and signing secret

### Setup (Stream Mode -- Recommended)

1. Go to the [DingTalk Open Platform](https://open.dingtalk.com/), create an application (or use an existing one).
2. Enable the **Robot** capability and note the **App Key** (Client ID) and **App Secret** (Client Secret).
3. Set environment variables:

```bash
export DINGTALK_APP_KEY=your-app-key          # Environment variable
export DINGTALK_APP_SECRET=your-app-secret
librefang vault set DINGTALK_APP_KEY          # Encrypted vault (recommended)
librefang vault set DINGTALK_APP_SECRET
```

4. Add to config:

```toml
[channels.dingtalk]
receive_mode = "stream"
app_key_env = "DINGTALK_APP_KEY"
app_secret_env = "DINGTALK_APP_SECRET"
default_agent = "assistant"
```

5. Restart the daemon.

### Setup (Webhook Mode)

1. In DingTalk, open a group chat and go to **Group Settings > Smart Group Assistant > Add Robot**.
2. Choose **Custom Robot** and enable the **Security Setting** (sign mode recommended).
3. Note the webhook access token and signing secret.
4. Set environment variables:

```bash
export DINGTALK_ACCESS_TOKEN=your-token       # Environment variable
export DINGTALK_SECRET=your-signing-secret
librefang vault set DINGTALK_ACCESS_TOKEN     # Encrypted vault (recommended)
librefang vault set DINGTALK_SECRET
```

5. Add to config:

```toml
[channels.dingtalk]
receive_mode = "webhook"
access_token_env = "DINGTALK_ACCESS_TOKEN"
secret_env = "DINGTALK_SECRET"
default_agent = "assistant"
```

6. Restart the daemon.

### How It Works

In **stream mode**, the adapter registers with the DingTalk gateway API (`/v1.0/gateway/connections/open`) to obtain a WebSocket endpoint and ticket. It then maintains a persistent WebSocket connection, receiving bot message callbacks and responding with ACK frames. Heartbeat ping/pong is handled automatically.

In **webhook mode**, the adapter receives messages via HTTP callback on the shared API server at `/channels/dingtalk/webhook`. Each request is verified using HMAC-SHA256 signature with the configured secret. Responses are sent back through the DingTalk webhook URL with a signed timestamp.

## QQ

### Prerequisites

- A QQ Bot registered on the [QQ Bot Platform](https://bot.q.qq.com/wiki/)
- An App ID and token

### Setup

1. Go to the [QQ Bot Platform](https://bot.q.qq.com/) and register a bot application.
2. Note the App ID and token.
3. Set environment variables:

```bash
export QQ_APP_ID=your-app-id                 # Environment variable
export QQ_TOKEN=your-bot-token
librefang vault set QQ_APP_ID                # Encrypted vault (recommended)
librefang vault set QQ_TOKEN
librefang config set-key qq                   # .env file
# Or set via dashboard "Set API Key" button   # secrets.env
```

4. Add to config:

```toml
[channels.qq]
app_id_env = "QQ_APP_ID"
token_env = "QQ_TOKEN"
default_agent = "assistant"
```

5. Restart the daemon.

### How It Works

The QQ adapter connects to the QQ Bot API via WebSocket for real-time message delivery. It authenticates using the app credentials, listens for message events in authorized guilds and channels, and sends responses through the QQ Bot REST API.

## WeCom

### Prerequisites

- A WeCom (WeChat Work) enterprise account with an **intelligent bot** created from the [WeCom Admin Console](https://work.weixin.qq.com/)

### Setup

1. In the WeCom admin console, go to **Apps > Create App** and select **Intelligent Bot**.
2. Note the **Bot ID** and **Secret** from the bot settings page.
3. Set the secret environment variable:

```bash
export WECOM_BOT_SECRET=your-bot-secret       # Environment variable
librefang vault set WECOM_BOT_SECRET          # Encrypted vault (recommended)
librefang config set-key wecom                # .env file
# Or set via dashboard "Set API Key" button   # secrets.env
```

4. Add to config:

```toml
[channels.wecom]
bot_id = "YOUR_BOT_ID"
secret_env = "WECOM_BOT_SECRET"
# account_id = "my-bot"          # Optional: unique ID for multi-bot routing
# default_agent = "assistant"    # Optional: route messages to specific agent
```

5. Restart the daemon.

### How It Works

The WeCom adapter connects to `wss://openws.work.weixin.qq.com` via a persistent WebSocket. It authenticates using the Bot ID and a signed token, then subscribes to incoming messages via `aibot_subscribe`. User messages arrive as `aibot_msg_callback` JSON frames and replies are sent back as `aibot_send_msg` frames. No public HTTP endpoint or AES encryption is required.

## Ntfy

### Prerequisites

- Access to an [ntfy](https://docs.ntfy.sh) server (self-hosted or the public instance at `ntfy.sh`)

### Setup

1. Choose a topic name on your ntfy server.
2. Set environment variables:

```bash
export NTFY_URL=https://ntfy.sh               # Environment variable
export NTFY_TOPIC=my-librefang-topic
# export NTFY_TOKEN=your-access-token         # Optional
librefang vault set NTFY_URL                  # Encrypted vault (recommended)
librefang vault set NTFY_TOPIC
# librefang vault set NTFY_TOKEN              # Optional
librefang config set-key ntfy                 # .env file
# Or set via dashboard "Set API Key" button   # secrets.env
```

3. Add to config:

```toml
[channels.ntfy]
url_env = "NTFY_URL"
topic_env = "NTFY_TOPIC"
# token_env = "NTFY_TOKEN"   # optional
default_agent = "assistant"
```

4. Restart the daemon.

### How It Works

The ntfy adapter subscribes to a topic on the ntfy server using Server-Sent Events (SSE) for real-time message delivery. Incoming messages are routed to the configured agent, and responses are published back to the topic via the ntfy REST API.

## Gotify

Gotify runs as an out-of-process Python sidecar adapter (`librefang.sidecar.adapters.gotify`, ships in `librefang-sdk`).

### Prerequisites

- A Gotify server instance.
- An **application** token (used to publish) and a **client** token (used to subscribe via WebSocket) — both created in the Gotify web UI.

### Setup

1. In the Gotify web UI, create an Application and copy its token (this becomes `GOTIFY_APP_TOKEN`); create a Client and copy its token (this becomes `GOTIFY_CLIENT_TOKEN`).
2. Open the dashboard Channels page, pick Gotify, and fill in the server URL and both tokens — the dashboard writes `GOTIFY_APP_TOKEN` / `GOTIFY_CLIENT_TOKEN` to `~/.librefang/secrets.env` and `GOTIFY_SERVER_URL` to the `[sidecar_channels.env]` table in `~/.librefang/config.toml`. Equivalent manual config:

```toml
[[sidecar_channels]]
name = "gotify"
command = "python3"
args = ["-m", "librefang.sidecar.adapters.gotify"]
channel_type = "gotify"
[sidecar_channels.env]
GOTIFY_SERVER_URL = "https://gotify.example.com"
# GOTIFY_ACCOUNT_ID = "prod"    # optional, multi-bot routing key
```

3. The daemon supervises the sidecar — no manual restart is needed; the kernel's hot-reload spawns or replaces the subprocess on config change.

### How It Works

The Gotify sidecar connects to `wss://{server}/stream?token={client_token}` to receive Gotify push messages in real time and POSTs back to `{server}/message?token={app_token}` to publish. The WebSocket client is a stdlib-only RFC 6455 reader on `socket` + `ssl` (no third-party dependency).

## Nextcloud Talk

Nextcloud Talk is provided by the Python sidecar adapter (`librefang.sidecar.adapters.nextcloud`). The in-process `[channels.nextcloud]` config block was removed in the sidecar migration. Declare Nextcloud as a `[[sidecar_channels]]` entry instead.

### Prerequisites

- A Nextcloud instance with the Talk app installed (see [Nextcloud Talk API docs](https://nextcloud-talk.readthedocs.io/))
- An app password or bot user account
- Python 3.8+ on the daemon host (the sidecar runs as `python3 -m librefang.sidecar.adapters.nextcloud`)

### Setup

1. In Nextcloud, go to **Settings > Security > Devices & sessions** and create an app password (or create a dedicated bot user).
2. Add the sidecar entry to your `config.toml`:

```bash
[[sidecar_channels]]
name = "nextcloud"
command = "python3"
args = ["-m", "librefang.sidecar.adapters.nextcloud"]
channel_type = "nextcloud"
[sidecar_channels.env]
NEXTCLOUD_SERVER_URL = "https://cloud.example.com"
# NEXTCLOUD_ROOMS = "abc123,def456"           # optional; empty = all joined
# NEXTCLOUD_ACCOUNT_ID = "prod"               # optional, multi-bot routing key
# NEXTCLOUD_POLL_INTERVAL_SECS = "3"          # optional, default 3, floor 1
```

3. Put `NEXTCLOUD_TOKEN` (the app password) in `~/.librefang/secrets.env` — the dashboard's Channels page writes it there when you fill the Nextcloud form.
4. Restart the daemon.

### How It Works

The sidecar polls `GET /ocs/v2.php/apps/spreed/api/v1/chat/<token>?lookIntoFuture=1` per room at the configured interval (default 3 s) using Bearer auth plus the mandatory `OCS-APIRequest: true` header. Slash-command bodies (`/cmd args`) become `Command` events; everything else is plain text. Outbound replies go through `POST /ocs/v2.php/apps/spreed/api/v1/chat/<token>` with a form-encoded body and thread correctly via the `replyTo` parameter when the inbound message lived inside a thread (improvement over the former Rust adapter, which never sent `replyTo`).

## Webhook

### Prerequisites

- A target URL that accepts HTTP POST requests (no external service registration needed)

### Setup

1. Choose a URL where webhook payloads should be sent.
2. Choose a shared secret for HMAC-SHA256 signature verification.
3. Set environment variables:

```bash
export WEBHOOK_URL=https://your-service.example.com/hook  # Environment variable
export WEBHOOK_SECRET=your-shared-secret
librefang vault set WEBHOOK_URL               # Encrypted vault (recommended)
librefang vault set WEBHOOK_SECRET
librefang config set-key webhook              # .env file
# Or set via dashboard "Set API Key" button   # secrets.env
```

4. Add to config:

```toml
[channels.webhook]
url_env = "WEBHOOK_URL"
secret_env = "WEBHOOK_SECRET"
default_agent = "assistant"
```

5. Restart the daemon.

### How It Works

The Webhook adapter provides a generic bidirectional HTTP integration. Incoming messages are received as HTTP POST requests on the shared API server at `/channels/webhook/webhook`, verified via HMAC-SHA256 using the shared secret. Responses are delivered as HTTP POST payloads to the configured URL, also signed with HMAC-SHA256.

### `deliver_only` Pass-through Mode

The webhook channel can also run as a zero-LLM pass-through: incoming payloads bypass the agent loop entirely (no sanitiser, no rate limiter, no agent lookup) and forward straight to a target channel. Useful for push notifications and out-of-band alerts where you only need the message to land in Telegram / Slack / etc.

```toml
[channels.webhook]
enabled = true
deliver_only = true
deliver = "telegram:123456789"     # required when deliver_only is true
```

When `deliver_only = true` but `deliver` is unset, librefang emits a startup `warn` and the webhook stays inactive. See [Webhook `deliver_only` Mode](/integrations/channels#webhook-deliver_only-mode) on the channels overview for the internal `__deliver_only__` / `__deliver_target__` metadata signalling.
